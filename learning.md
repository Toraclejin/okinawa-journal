## 모바일에서 왜 그렇게 자꾸 깨졌나

제일 많은 시간을 쏟은 구간. 총 6가지 함정을 밟음.

### 1. 큰 글자를 숫자로 못 박아놓음

**문제**: CSS에 "타이틀은 280픽셀 크기"라고 숫자를 못 박아놓았음.
**비유**: 티셔츠 공장에서 XL 사이즈만 만들어놓고, S 사이즈 손님이 오면 옷에 몸을 우겨넣는 꼴. 옷은 펑퍼짐, 사람은 불편.
**결과**: 모니터에선 멋있게 큰 글자가, 아이폰에선 화면 밖으로 삐져나가서 잘려 보임.
**해결**: `clamp(최소, 화면비례, 최대)` 쓰기. "작은 화면에선 40px, 큰 화면에선 200px, 사이는 화면 비례로 자동 계산". 옷에 비유하면 신축성 있는 레깅스.

### 2. "반반씩 나누자"가 반반이 아니었던 문제

**문제**: CSS 그리드에서 `1fr 1fr`이라고 적으면 "반반씩 나눠라" 인 줄 알았는데, 아니었음.
**비유**: 형제 둘이 방을 반반 나누기로 했는데, 한 명이 피아노를 들여놓음. 피아노가 너무 커서 자기 방을 밀어냄. 다른 형제 방은 좁아짐.
**실체**: `1fr`은 "남는 공간을 1몫씩 나눠 가져라"인데, **속내는 "내 안의 내용물보다는 절대 작아지지 않음"**. 그래서 큰 글자가 든 칸이 부풀어오르면 다른 칸이 찌그러짐.
**해결**: `minmax(0, 1fr)`로 바꾸기. "내용물이 아무리 커도 1몫만큼만 가져가라"는 엄격한 반반 규칙. CSS 그리드 쓸 때 필수 공식.

### 3. 이름표를 엉뚱한 사람 명찰 안쪽에 붙임

**문제**: CSS 그리드는 "A 칸에 뭘 놓을지" 지정해야 하는데, 그 지정을 **엉뚱한 깊이**에 걸어놨음.
**비유**: 회사에서 "박 부장은 4번 자리"라고 정해놓고, 막상 박 부장의 **책상 서랍 안**에 '4번 자리' 이름표를 붙여둔 꼴. 회사에선 박 부장이 어디 앉을지 모르니까 아무데나 앉혀버림.
**실체**: HTML 구조가 `<바깥박스><안박스>내용</안박스></바깥박스>`인데, `grid-area`를 안박스에만 붙였음. CSS 그리드는 **직접 자식**(바깥박스)만 보는데.
**해결**: 바깥박스에 `grid-area`를 붙이기. 혹은 `:nth-child(1)` 같은 순서 지정도 OK.

### 4. 페인트통 라벨만 바꿨더니 벽이 안 보임

**문제**: "우리 회사는 이제 파란색만 쓴다"라는 의도로 원래 있던 '코랄색' 페인트통에 파란색을 부었음.
**비유**: 그런데 옛날 설계도엔 "벽은 코랄 페인트, 그 위에 코랄 테두리 원을 그려라"라고 되어 있었음. 설계도 그대로 쓰니까 **파란 벽에 파란 원** = 원이 안 보임. Day 3 타이틀 글자 위에 같은 색 원이 겹쳐서 글자가 사라졌던 이유가 이거.
**교훈**: 색 이름을 하나로 통일할 땐, **그 색이 어디서 어떻게 쓰이는지 전부** 확인해야 함. 그냥 페인트통만 바꾸면 과거의 디자인 의도가 깨짐.
**다음엔**: 색 이름을 "파랑", "코랄"처럼 색으로 짓지 말고 **용도**로 지을 것. 예: "배경색", "강조색", "테두리색". 그러면 어디 쓰는지 이름만 봐도 앎.

### 5. 아이폰이 옛날 것을 기억함

**문제**: 코드를 배포하고 아이폰에서 새로고침해도 이전 버전이 보임.
**비유**: 냉장고가 일주일 전 음식을 너무 오래 기억해서, 오늘 새로 넣은 음식을 달라고 해도 자꾸 옛날 것만 꺼내줌.
**실체**: 아이폰 Safari는 웹페이지의 CSS·JS를 오래 캐시에 붙들어둠. 게다가 아이폰의 모든 브라우저(크롬, 파이어폭스 포함)는 다 Safari 엔진을 씀. 즉 크롬 깔아도 똑같음. (애플 정책)
**해결**: HTML에 "이 페이지는 캐시하지 말아달라"는 메타 태그 추가. 그리고 사용자한테는 "주소창 내려서 pull-to-refresh 하면 강제 새로고침됨"이라고 알려주기.

### 6. 항상 떠 있는 단추가 내용을 가림

**문제**: 화면 하단에 고정된 툴바(스크롤 버튼, 다운로드 등)가 페이지 맨 아래 "다음 페이지" 버튼을 덮어버림. 눌러지지 않음.
**비유**: 식당 테이블 위에 메뉴판을 풀로 고정해놨는데, 음식 접시 놓을 자리랑 겹침. 음식 나오면 메뉴판에 가려져서 뭘 먹는지 안 보이고, 먹기도 불편.
**해결**: "다음 페이지 버튼이 화면에 보일 때만 툴바를 자동으로 아래로 내리기". 음식 나오면 메뉴판이 알아서 비켜주는 식.
**기술명**: `IntersectionObserver` (요소가 화면에 보이는지 감시하는 브라우저 내장 기능).

## 콘텐츠 검증 — 링크가 엉뚱한 곳으로 가던 이유

### "Sunset Beach"가 호텔로 연결됐던 사건

**문제**: 가족들이 SUNSET 카드 누르면 구글맵이 "SUNSET 호텔"을 보여줌. 우리는 해변을 기대했는데.
**비유**: 외국 손님에게 "김씨 아저씨 가게 어디냐"고 물으면 한국에 김씨 아저씨가 너무 많아서 엉뚱한 가게로 안내되는 것.
**실체**: "Sunset Beach"는 전 세계 수백 곳에 있음. 그냥 그 단어만 던지면 구글이 아무거나 고름.
**해결**: **공식 명칭으로 검색어 고정**. "Chatan Park Sunset Beach" 처럼 정부에서 지정한 정확한 이름으로. 코드에 장소별 검색어 사전(`PLACE_QUERIES`)을 만들어서 클릭 시 정확한 검색어가 전송되도록.
**교훈**: 검색 엔진에 "알아서 찾아줘" 하지 말고, **정확한 이름을 직접 지정**. 게으르게 쓰면 게으른 결과 나옴.

### "근처 소바집" 같은 애매한 표현 금지

**문제**: 처음엔 "추라우미 근처 소바집" 이라고 막연하게 적어놨음. 링크 걸 수도 없고, 가족이 봐도 어딘지 모름.
**교훈**: 장소는 **고유명사**로. "1905년 창업 Kishimoto Shokudo" 정도는 돼야 검증도 되고 구글맵도 찾아냄.

### 주소는 건물명·번지까지

**처음**: "Depot Island Seaside 3F, 호텔 도보 3분"
**정확**: "depoisland seaside 3F · 미하마 9-21 · 호텔 바로 옆"
호텔이 미하마 9-22, Calif Kitchen이 미하마 9-21. **옆 건물**. 처음 썼던 "도보 3분"은 틀림.

## 보안 — "가족용인데 이런 걸 왜 해?"

### 하나밖에 없는 공격 경로

이 앱에는 "↓ 가져오기" 버튼이 있다. JSON 파일 하나 받으면 여행 기록 전체를 그걸로 덮어씀. 편리하지만, 바로 여기가 유일한 공격 창구였다.

**문제**: 원래 코드는 받은 JSON 안의 사진 주소를 HTML 문자열에 그대로 갖다 붙였음.
**비유**: 택배 상자 안에 뭐가 들었는지 안 보고 그냥 열어서 식탁에 통째로 쏟아버리는 것. 정상 택배면 괜찮은데, 가끔 폭탄이 들어있다.
**실체**: 교묘하게 만든 "사진 주소" 문자열이 HTML 속성의 따옴표 틈을 뚫고 **임의의 자바스크립트로 둔갑** 가능. "가족이 보낸 백업이야~" 하면서 악성 JSON을 피해자가 가져오면 그 사람 브라우저에서 실행됨.
**실제 영향도**: 이 앱엔 로그인·개인정보·돈 없음. 빼내봐야 남의 여행 메모 정도. 실제 위협은 낮았음.

### 4겹 방어로 고침

비유: **택배 검수 규칙**.

1. **받은 상자 그대로 쓰지 말고, 내용물 하나씩 꺼내서 새 상자에 담기**
   `sanitizeImportedState()` 함수가 허용된 필드만 옮기고 나머지는 폐기. 이름이 이상한 필드, 값이 이상한 필드 전부 버림.

2. **"사진"이라는 상자는 진짜 사진인지 포맷 검사**
   사진 데이터는 반드시 `data:image/...` 형식으로 시작해야 하고, 용량 상한도 있음. 아니면 거절.

3. **HTML은 문자열 합치기 금지, 브라우저 API로 만들기**
   원래 코드: `"<img src='" + 받은데이터 + "'>"` ← 이 방식이 위험
   고친 코드: `img = createElement; img.src = 받은데이터` ← 브라우저가 자동으로 따옴표 이스케이프
   비유: **손님이 써준 메모를 메뉴판에 풀로 바로 붙이지 말고, 메뉴판 시스템에 입력.** 시스템이 알아서 이상한 문자 처리함.

4. **특수 키는 바로 거절**
   JSON 안에 `__proto__` 같은 자바스크립트 특수 키가 있으면 즉시 폐기. 놔두면 앱 전체 동작이 오염될 수 있는 열쇠.

### 왜 고쳤나 — "가족만 쓰는데?"

진짜 이유는 **도쿄 여행 개인용 버전 복제 예정**이라서. 지금 원본에 있는 구멍은 복사본에도 그대로 따라감. 개인용은:
- 본인만 쓰니까 공격 기회는 여전히 적음
- 대신 들어갈 데이터가 더 민감해질 수도 있음 (예약번호, 개인 일기, 지출)
- 혹시 나중에 링크를 블로그·SNS에 흘리면 순식간에 공개 상태

**교훈**: "지금은 안 위험해"는 맞지만, **코드는 언젠가 복제·공유됨**. 원본에 구멍 있으면 사본도 그대로 뚫린 채로 복제됨. 지금 10분 투자 = 미래의 모든 사본이 안전.
**다음엔**: 새 여행지 버전 만들 때 sanitizer 먼저 옮기고 콘텐츠 작업. 순서 바꾸지 말 것. 새 state 필드 추가할 때마다 sanitizer에 대응 블록 같이 추가.

## 숙소를 잘못 알고 있었던 사건 — "현지에선 못 고친다"

### 발단

매거진 거의 완성된 시점, 가족과 일정 확인하다가 알아차림: **4박 내내 차탄 Vessel 호텔에 있는 줄 알았는데 실제로는 북부 Orion 2박 + 차탄 Vessel 1박**이었음. 매거진은 전부 "호텔이 아메리칸 빌리지 안이라 도보 1분..." 전제로 쓰여있었음.

**비유**: 결혼식 청첩장을 다 돌렸는데 예식장 주소가 틀린 거랑 비슷함. 한 번 인쇄·배포되면 수정 못 하니 그 전에 잡아야 함.

### 왜 이게 큰 문제였나

"현지에선 수정 불가"라는 제약이 모든 걸 바꿈:
- 여행 중 "이거 주소 틀렸네" 발견 → 당황 → 검색 → 시간 낭비
- 구글맵 링크가 엉뚱한 곳으로 → 엄마가 혼자 길 잃음
- 식당 영업시간 착오 → 문 닫힌 가게 앞에서 가족 전체 당황

매거진은 **"정적인 안내서"**라 소프트웨어처럼 핫픽스가 안 됨. 종이에 박힌 글자와 똑같음.

6. **Google Maps 쿼리** — 실제로 쳐서 정확한 핀이 나오는지 확인

**비유**: 건설 현장의 측정 원칙 **"Measure twice, cut once"**. 한 번 잘라낸 자재는 못 되돌리는 거랑 같음. 매거진도 배포 후엔 못 되돌림.

### 교훈 — 한 줄

> **"현지 수정 불가"라는 제약은 제약이 아니라 규율이다.**

소프트웨어는 배포 후에도 고치지만, 여행 매거진은 **가족이 공항에 서있는 순간 고정된다**. 이 긴장감이 오히려 **검증의 표준을 높인다**. 다음에 도쿄 버전 만들 때는 처음부터 이 검증 체크리스트를 밟고 시작할 것.

### 다음엔 뭘 다르게

1. **첫 커밋 전에 예약증 다 긁어서 CLAUDE.md에 박아두기** — 숙소·항공·렌터카 모두 명시. 글자로 박혀있으면 전제가 안 꼬임.
2. **장소 추가할 때 "✅ 공식 확인" 표시** — 안 붙은 장소는 PR에 올리지 말 것.
3. **에이전트에 검증 아웃소싱** — 팩트 체크는 집중력 소모 크니 웹 검색 에이전트에 "이 10가지를 공식 URL로 확인해서 표로 줘" 시키고, 돌아온 결과만 보면 됨.

## 두 세션이 같은 벽을 칠하면 — 도쿄 폴더 충돌 사건

### 무슨 일이 있었나

도쿄 매거진을 만들겠다고 오키나와 채팅방에서 도쿄 폴더에 셸을 만들기 시작했다(오키나와 6500줄 코드 복사 + 식별자만 도쿄로 교체). 17:25–17:28에 작업 완료.

근데 같은 시간, **다른 채팅방**에서 도쿄 PRD v0.2 기반으로 v1을 빌드하고 있었다 (single-page Quick Capture · IndexedDB · 1700줄 슬림 버전). 17:41에 그쪽 세션이 `index.html`을 v1으로 덮어씀.

결과:
- 내가 만든 셸 → 사라짐 (덮어쓰기 당함)
- 시간 낭비 + 사용자 혼선
- 다행히 git이 따로 있는 폴더가 아니라 충돌 자체는 안 났지만, 큰 작업이었으면 한 쪽 일이 통째로 날아갈 수 있었음

### 비유

**이사 중인 집에 짐꾼 두 팀이 따로 와서 같은 가구를 옮긴 셈**. 둘 다 일을 하고 있었지만, 한 팀은 다른 팀이 이미 옮기고 있다는 걸 모름. 결국 한 팀의 작업은 의미 없어짐.

또는 **같은 벽에 두 사람이 다른 색 페인트를 동시에 칠하는 것**. 마지막에 칠한 사람이 이김. 먼저 칠한 사람의 작업은 그대로 사라짐.

### 왜 이게 일어났나

- **폴더 단위 잠금 메커니즘이 없음** — git이 있는 곳이면 동시 편집이 충돌로 잡히지만, git 없는 폴더는 그냥 마지막 저장이 이김
- **Claude Code 세션 간 통신이 없음** — 다른 세션이 뭘 하고 있는지 서로 모름
- **사용자가 두 채팅방에서 같은 폴더를 열어줬음** — 이게 발생 조건

### 교훈

1. **한 폴더 = 한 세션**. 부모 README에 명시 룰 추가됨 ("⚠️ 여러 Claude Code 세션이 같은 매거진 폴더를 동시에 작업하지 말 것").
2. **다른 세션이 진행 중인 폴더에는 직접 파일 안 건드리고 명세서로 넘기기**. 코드 직접 수정 = 충돌. 대신 다음 패턴:
   - 부모 폴더에 `XXX_TRANSPLANT_TO_YYY.md` 같은 이름의 명세서 파일 작성
   - 명세에는 "어디 코드를 어떻게 변형해서 가져갈지" 자세히
   - 다른 세션이 이 파일 읽고 작업
   - 결과는 다른 세션이 자신의 `learning.md` / `CLAUDE.md`에 기록
3. **명세서를 부모 폴더에 두는 이유**: 두 폴더(소스·대상) 모두에서 참조 가능. 어느 한 폴더에만 두면 다른 쪽 세션이 못 보거나 cwd 충돌.
4. **시작 전 modification time 확인**. 폴더에 누군가 최근 작업한 흔적이 있으면(파일 timestamp가 분 단위로 새것이면) 일단 멈추고 상황 파악.

### 다음에 비슷한 일 할 때

**A 세션이 B 폴더 작업에 기여하고 싶으면**:
1. B 폴더 파일 직접 수정 ❌
2. 부모 폴더에 명세서 파일 작성 ✅
3. 사용자에게 "B 채팅방에 이 파일 가리키고 작업 시켜" 안내
4. B 세션이 적용 후 명세서 진행 상황 체크박스 갱신
5. 나중에 명세서 자체는 archive 폴더로 이동하거나 그대로 두고 참고용

**예시 (실제)**: `UI_TRANSPLANT_TO_TOKYO.md` (2026-04-25 작성) — 오키나와의 인터랙션·툴 13개를 도쿄 v1에 가져갈 때 어떻게 변형할지 Tier 1/2/3로 분류한 명세서.

## 왜 React 안 썼냐

간단하게 비교해봄:

| 상황 | vanilla (지금 방식) | React |
|---|---|---|
| 한 파일로 끝내고 싶음 | 완벽 | 여러 파일 + 빌드 단계 필요 |
| 가족이 수정하다 깨뜨릴 여지 | 거의 없음 | package.json 잘못 건드리면 사이트 다운 |
| 호텔 Wi-Fi 불안정해도 작동 | 한 번 받으면 오프라인 OK | 번들 받아야 함 |
| 새 여행지 버전 만들기 | 파일 복사 + 내용 교체 | 컴포넌트 재사용 가능 |

**비유**: 동네 분식집 차리는데 공장을 지을 필요 없음. 분식집은 분식집답게 간단한 게 답. 나중에 "전국 체인 100개 내겠다" 하면 그때 공장 고민.

지금 이 프로젝트는 **딱 분식집 규모**. React는 과함.

## 다음에 비슷한 거 만들 때 다르게 할 5가지

1. **처음부터 아이폰 화면으로 테스트**
   지금은 큰 모니터에서 만들고 마지막에 아이폰 확인했더니 패치의 패치가 이어짐. 다음엔 아이폰 뷰로 먼저 본 후에 모니터 버전을 덧붙이기. (모바일 퍼스트)

2. **색 이름을 용도로 짓기**
   `--coral`, `--blue` 같은 색깔 이름 말고 `--배경-강조`, `--글자-진한`, `--테두리-부드러움` 식으로. 이름만 봐도 어디 쓰는지 앎.

3. **장소 데이터를 따로 분리**
   지금은 HTML 안에 장소 이름이 여기저기 박혀 있어서, 주소 하나 고치려면 여러 군데 수정해야 함. 다음엔 `places.json` 같은 데이터 파일 하나에 모으고, HTML은 JS가 그려주는 방식.

4. **Claude Preview로 수시 확인**
   배포 전에 미리 375×812 (아이폰 크기) 뷰포트로 스크린샷 찍어서 확인. 5초 걸리는데 나중에 쏟아질 버그 리포트 수십 개를 미리 막음.

5. **공식 명칭으로 장소 검증**
   장소 이름 쓸 땐 반드시 공식 사이트·트립어드바이저에서 한 번 더 확인. "Sunset Beach 사건"처럼 대충 쓰면 링크가 엉뚱한 곳으로 가고 가족이 혼란.

## 사용자가 제일 많이 해준 피드백

> **"모바일로 다 잘리는데?"**
→ 내가 또 아이폰 확인 안 하고 배포했다는 신호. 이 문장 들을 때마다 자동 리플렉스로 "프리뷰 켜서 모바일 스크린샷 먼저" 하는 습관 생김.

> **"이거 정확한 거 맞아?"**
→ 장소 정보는 한 번 더 검증. 특히 "도보 몇 분"은 실제 지도로 재봐야 정확.

> **"너무 길어서 다 안 읽혀"**
→ 모바일에서 긴 본문은 자동으로 3줄까지 접고 "+ 더보기" 버튼. 관심 있는 사람만 펼침.
